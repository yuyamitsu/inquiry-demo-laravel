# チケット管理ツール比較と問い合わせ管理Demoへの反映方針

## 目的

問い合わせ管理Demoを、単なる問い合わせフォームではなく、Redmine、Backlog、Jira などのチケット管理ツールを参考にしながら、どのような機能を持たせるべきか整理する。

特に、管理する規模が大きくなった場合に必要になる以下の観点を整理する。

* プロジェクト
* ユーザー
* ロール
* グループ
* タスク
* 子タスク
* 関連付け
* コメント履歴
* ステータス管理
* 担当者
* 優先度
* 期限

本Demoでは、Redmine、Backlog、Jira のような大規模チケット管理システムをそのまま再現するのではなく、**中規模の問い合わせ・チケット管理システム** を想定し、必要な機能を段階的に実装する方針とする。

---

## 現在の問い合わせ管理Demoの位置づけ

現在のDemoでは、問い合わせ1件を1つのチケットのように扱っている。

```text
問い合わせ = チケット
管理者 = 全問い合わせを確認・更新するユーザー
一般ユーザー = 自分の問い合わせを確認するユーザー
ステータス = 未対応 / 対応中 / 回答済み / クローズ
コメントスレッド = 管理者と一般ユーザーのやり取り
変更履歴 = 管理項目の更新履歴
```

本Demoは、単なる問い合わせフォームではなく、**中規模の問い合わせ・チケット管理システム** を想定して設計している。

中規模とは、問い合わせ件数や対応者が増え、以下のような管理が必要になる段階を想定する。

* 問い合わせごとに担当者を設定したい
* 優先度を付けて対応順を判断したい
* 対応期限を管理したい
* 管理者と一般ユーザーのやり取りをコメント形式で残したい
* 変更履歴を追跡したい
* ステータスや条件で検索・絞り込みしたい
* 管理者と一般ユーザーで利用できる画面を分けたい
* 将来的にプロジェクトやグループ単位で管理を拡張できるようにしたい

現時点では、中規模管理の土台として以下を実装済み。

* 管理者ログイン
* 一般ユーザーログイン
* role による管理者 / 一般ユーザーの分岐
* 管理者Middlewareによる `/admin` 配下のアクセス制御
* 問い合わせ登録
* 一般ユーザーによる自分の問い合わせ一覧・詳細確認
* 管理者による全問い合わせ一覧・詳細確認
* ステータス更新
* コメントスレッド
* 管理者・一般ユーザーのコメント投稿
* 担当者設定
* 優先度設定
* 対応期限設定
* 変更履歴
* 更新者表示
* 検索・絞り込み
* ページネーション
* 件数サマリー

## 規模別に必要な機能

### 小規模

小規模な問い合わせ管理は、会社の問い合わせフォームのような **多対1** の構成と考える。

```text
多くの利用者
↓
1つの問い合わせ窓口
↓
少人数の管理者が対応
```

例：

* 会社HPの問い合わせフォーム
* 小規模店舗の問い合わせフォーム
* 個人事業の相談フォーム
* 小規模サービスの簡易サポート窓口

少人数で問い合わせを管理する場合は、最低限以下の機能があれば運用できる。

* 問い合わせ登録
* 管理者一覧
* 詳細確認
* ステータス管理
* 管理者返答
* 一般ユーザーによる自分の問い合わせ確認

小規模では、担当者や優先度、期限管理がなくても、管理者が目視で対応状況を把握できる。

ただし、問い合わせ件数が増えたり、複数人で対応するようになると、対応漏れや優先順位の判断が難しくなる。

---

### 中規模

本Demoでは、この中規模の問い合わせ管理を主な想定範囲とする。

中規模は、利用者が複数いて、対応する管理者・担当者も複数いる **多対複数** の構成と考える。

```text
多くの利用者
↓
問い合わせ窓口
↓
複数の管理者・担当者で対応
```

例：

* 社内ヘルプデスク
* ITサポート窓口
* 部署内の問い合わせ管理
* 中小企業の顧客サポート
* 学校・団体の問い合わせ管理

中規模では、問い合わせ件数や対応者が増えるため、以下の機能が重要になる。

* コメントスレッド
* 担当者
* 優先度
* 対応期限
* ステータス管理
* 管理者ロールの整理
* 検索条件の追加
* 変更履歴
* 更新者表示

この段階では、問い合わせ1件に対して複数回のやり取りが発生するため、`admin_reply` のような1つの返答欄だけでは不足する。

そのため、本Demoでは `inquiry_comments` テーブルを追加し、問い合わせ1件をスレッドとして扱えるようにしている。

また、問い合わせごとに担当者、優先度、対応期限を持たせることで、複数人で対応する場合でも、誰が・いつまでに・どの優先度で対応するかを確認できるようにしている。

本Demoでは、Redmine、Backlog、Jira のような大規模チケット管理の全機能を実装するのではなく、中規模運用に必要な機能を優先して実装している。

---

### 大規模

大規模なチケット管理は、利用者・担当者・部署・プロジェクトが複数存在する **多対多 + 複数プロジェクト** の構成と考える。

```text
多くの利用者
↓
複数の窓口・複数プロジェクト
↓
複数部署・複数チームで対応
```

例：

* 大企業の社内ITチケット管理
* 複数部署をまたぐ業務改善依頼管理
* 開発会社の複数案件管理
* SaaS企業のサポート・開発連携
* Redmine / Backlog / Jira 的な課題管理

複数部署、複数案件、複数チームで管理する場合は、以下が重要になる。

* プロジェクト
* グループ
* プロジェクトごとのユーザー権限
* 親タスク
* 子タスク
* 関連チケット
* 横断検索
* ダッシュボード
* 通知
* 操作ログの詳細化

この段階では、問い合わせ管理というより、チケット管理・課題管理システムに近い構成になる。

大規模向け機能は将来的な拡張候補とし、現時点では中規模運用に必要な機能を優先する。

## 本Demoで実装した中規模向け機能

本Demoでは、中規模の問い合わせ・チケット管理を想定し、実運用で必要になりやすいものから段階的に追加した。

実装済みの主な機能は以下。

```text
1. コメントスレッド
2. 担当者
3. 優先度
4. 対応期限
5. 管理者Middleware
6. 検索条件の追加
7. 一般ユーザー一覧での進捗確認
8. 変更履歴
9. 更新者表示
```

### 実装した理由

中規模では、問い合わせ数や対応者が増えるため、単に問い合わせを一覧表示するだけでは管理が難しくなる。

特に重要なのは以下。

| 機能            | 理由                       |
| ------------- | ------------------------ |
| コメントスレッド      | 問い合わせ1件に対して複数回のやり取りを残すため |
| 担当者           | 誰が対応する問い合わせかを明確にするため     |
| 優先度           | 急ぎの問い合わせを判断しやすくするため      |
| 対応期限          | 対応漏れや遅延を防ぐため             |
| 管理者Middleware | 管理者権限を整理し、アクセス制御を明確にするため |
| 検索条件の追加       | 問い合わせ件数が増えても対象を探しやすくするため |
| 変更履歴          | いつ・誰が・何を変更したかを追跡するため     |

一方で、プロジェクト、グループ、子タスク、関連チケットは、大規模運用では重要だが、現時点の中規模Demoでは優先度を下げる。

そのため、本Demoではまず以下の構成を目標とした。

```text
問い合わせ
├── コメントスレッド
├── 担当者
├── 優先度
├── 対応期限
├── ステータス
└── 変更履歴
```

この構成により、単なる問い合わせフォームではなく、問い合わせをチケットとして継続的に管理できる中規模向けの構成に近づけている。

## コメントスレッド

初期実装では、`inquiries.admin_reply` に管理者返答を1つだけ保存する構成だった。

しかし、この方式では以下のようなやり取りには弱い。

```text
一般ユーザー：追加で質問
管理者：追加回答
一般ユーザー：確認しました
管理者：クローズしました
```

そのため、問い合わせ1件に対して複数のコメントを持てる構成に変更した。

追加したテーブルは以下。

```text
inquiry_comments
├── id
├── inquiry_id
├── user_id
├── body
├── created_at
└── updated_at
```

この構成により、問い合わせ1件をスレッドのように扱える。

### 管理者詳細画面

```text
問い合わせ内容

管理項目
- ステータス
- 担当者
- 優先度
- 対応期限

コメントスレッド
[利用者] 料金について教えてください
[運営] スタンダードプランは月額〇〇円です
[利用者] 年払いは可能ですか？
[運営] はい、可能です

コメント投稿フォーム

変更履歴
```

### 一般ユーザー詳細画面

```text

- 現在のステータス
- 担当者
- 優先度
- 対応期限

コメントスレッド
[利用者] 追加で質問です
[運営] 回答します

コメント投稿フォーム
```

## 担当者・優先度・対応期限

中規模管理では、問い合わせ件数や対応者が増えるため、問い合わせごとに以下の情報を持たせる。

```text
assignee_id
priority
due_date
```

### assignee_id

問い合わせを担当する管理者ユーザーを表す。

```text
assignee_id = 対応担当者
```

これにより、管理者一覧で「誰が対応する問い合わせか」を確認できる。

また、一般ユーザー側でも担当者を表示することで、自分の問い合わせを誰が確認しているか把握しやすくしている。

---

### priority

問い合わせの優先度を表す。

```text
低
中
高
緊急
```

これにより、対応順や重要度を判断しやすくする。

---

### due_date

対応期限を表す。

```text
due_date = いつまでに対応するか
```

これにより、対応漏れや対応遅れを防ぎやすくする。

## 管理者Middleware

管理者画面へのアクセス制御をController内だけで行うと、各処理に同じ判定を書く必要があり、管理しづらくなる。

そこで、`AdminMiddleware` を作成し、`/admin` 配下のルートに適用した。

```php
Route::middleware(['auth', 'admin'])
    ->prefix('admin')
    ->name('admin.')
    ->group(function () {
        // 管理者用ルート
    });
```

これにより、一般ユーザーが `/admin/inquiries` に直接アクセスした場合は、Controllerに到達する前に 403 で弾かれる。

### Middleware化の目的

* 管理者画面へのアクセス制御を一か所にまとめる
* Controller内の責務を軽くする
* 今後管理者機能が増えても、同じMiddlewareを使い回せるようにする

## admin_reply の扱い

初期実装では、管理者からの返答を `inquiries.admin_reply` に保存していた。

しかし、コメントスレッド機能を追加したことで、管理者と一般ユーザーのやり取りは `inquiry_comments` に保存する方針へ移行している。

### 現在の方針

現在は、既存機能を壊さないために `admin_reply` カラムは残している。

ただし、画面上のやり取りはコメントスレッドを使用し、`admin_reply` は基本的に使用しない。

今後、コメント機能が安定した段階で、`admin_reply` の削除または補助項目化を検討する。

## 今後の設計方針

本Demoでは、まず中規模な問い合わせ管理として必要な機能を優先して実装する。

中規模では、問い合わせ1件に対して複数回のやり取りが発生し、対応者も複数になる可能性がある。

そのため、以下を重視した構成にしている。

* コメントスレッドでやり取りを残す
* 担当者で責任範囲を明確にする
* 優先度で対応順を判断しやすくする
* 対応期限で対応漏れを防ぐ
* 管理者Middlewareで権限制御を明確にする
* 変更履歴で操作内容を追跡する

その後、必要に応じて以下を追加検討する。

* プロジェクト
* グループ
* 子タスク
* 関連チケット

これにより、単なる問い合わせフォームではなく、問い合わせをチケットとして管理できる中規模向けの構成に近づける。
